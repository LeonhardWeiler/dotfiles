1 · Open findings

SEC-1 high config/bash/.bashrc · config/claude/settings.json

Two settings compound into "no guard rails at all" for the agent that has write access to this machine. .bashrc defines alias claude='claude --dangerously-skip-permissions', so every invocation skips all permission prompts - and because the alias shadows the binary, there is no ordinary way to start it normally short of \claude or command claude. settings.json then adds "skipDangerousModePermissionPrompt": true, which also removes the one-time warning about running in that mode. On a box where ./install --sudoers installs %wheel ALL=(ALL:ALL) NOPASSWD: ALL, an agent mistake escalates to root without a single prompt. Neither setting is mentioned in README or CLAUDE.md, so it does not currently read as a deliberate decision.

Recommendation: keep plain claude intact and put the bypass behind a separate, explicitly-named alias (alias yolo='claude --dangerously-skip-permissions'). If the current behaviour is genuinely wanted, document it in the README next to the passwordless-sudo note so the combined blast radius is written down somewhere.
UX-1 medium config/dwl/config.h:113

Alt+Shift+Q calls quit and tears down the whole session immediately - no confirmation, everything unsaved is gone. It sits on the same modifier as, and one key row from, Alt+Shift+C (killclient), which is a key you press all day. Ctrl+Alt+Backspace (Terminate_Server) is a second unconfirmed way to do the same thing. Since dwl is started via exec dwl from ~/.bash_profile, quitting drops straight back to a bare tty1 shell.

Recommendation: move quit to a chord that cannot be hit by accident (e.g. add WLR_MODIFIER_CTRL), or route it through a confirmation with the rofi that is already installed: SHCMD("printf 'no\\nyes' | rofi -dmenu -p 'quit dwl?' | grep -qx yes && pkill dwl").
UX-2 medium config/dwl/config.h

Several capabilities are compiled in but unreachable, because no key is bound to them - the build itself warns about exactly these (setlayout / focusmon / tagmon / incnmaster / toggletag defined but not used):

  • layouts[] defines three layouts ([]= tile, ><> floating, [M] monocle) but nothing is bound to setlayout, so floating and monocle can never be selected. The comment in the file claims they "stay available", which only holds for per-window togglefloating and togglefullscreen.
  • monrules[] carries a rule for an external output (the NULL catch-all next to eDP-1), but focusmon/tagmon are unbound - with a second display attached there is no way to focus it or move a window to it.
  • incnmaster is unbound, so the master count is permanently pinned at 1.
  • There is no binding to move the focused window within the stack - only focusstack (Alt+j/k) and zoom.
Recommendation: decide per item - either bind them (Alt+t/f/m for the layouts, Alt+,/. for focusmon/tagmon, Alt+i/d for incnmaster) or delete the two unreachable layouts and the external monrules entry so config.h stops promising what it cannot do.
CQ-1 medium config/dwl/build-dwl:29-38

The build unconditionally runs git fetch --tags --depth 1 origin v0.8 before touching anything, and the script runs under set -eu - so if the remote is unreachable the whole ./install --dwl step aborts, even when a valid, pinned v0.8 checkout is already sitting in ~/.local/src/dwl. This is not theoretical: it happened during this review (codeberg.org answered HTTP 504), and the intended way to apply a config.h change was unavailable until the build was reproduced by hand. Since dwl is configured at compile time, this makes editing config.h dependent on a third party being up.

Recommendation: only fetch when the pinned tag does not already resolve locally (git -C "$SRC_DIR" rev-parse -q --verify "refs/tags/$DWL_VERSION" >/dev/null || git fetch …), or make the fetch non-fatal (|| echo "offline - using the local checkout") and let reset --hard do the rest.
CQ-2 medium install · setup/links.conf

There is no way to clean up links whose entry was removed from links.conf. unlink and status both iterate the current entries only, so a deleted line leaves its symlink behind forever - invisible to every command the script offers. Hit during this review: dropping the hyprlock entry left ~/.config/hypr/hyprlock.conf dangling and it had to be removed by hand. The same applies to the two commented-out wallpaper lines. An earlier clean command apparently existed but is gone.

Recommendation: reintroduce a prune command that walks the managed target roots and removes symlinks pointing into $BASEDIR that no entry claims any more - or, cheaper, have status report such orphans so they at least become visible.
DOC-1 medium README.md · config/dwl/README.md

The keybindings are documented nowhere. For a setup whose entire premise is "keyboard-driven", the only description of its primary interface is config/dwl/config.h itself. That matters more than usual here because the choices are unusual: MODKEY is Alt, not Super, while the screen lock is the only binding on Super (Super+L) - and there are exactly 6 tags, not the customary 9.

Recommendation: add a keybinding table to config/dwl/README.md (modifier, key, action), grouped the way config.h already groups them, and link it from the root README.
DOC-2 medium config/dwl/README.md:13-20,35

Two inaccuracies in the dwl README:

  • Line 35 says "Log out and pick dwl in ly" - there is no display manager. ly is gone; getty@tty1 autologins and ~/.bash_profile execs dwl. Following the instruction is impossible.
  • The file list names only patches/gaps.patch and patches/autostart.patch, but build-dwl:40 applies every patches/*.patch - so patches/attachbottom.patch is compiled in as well and is documented nowhere (CLAUDE.md has the same gap). New windows attaching at the bottom of the stack is user-visible behaviour.
Recommendation: replace the ly sentence with the actual apply path (rebuild, then log out of the session / reboot), and list all three patches with a one-line description each in both config/dwl/README.md and CLAUDE.md.
DOC-3 medium CLAUDE.md:115,135

CLAUDE.md lists vconsole among the config/ sources and vconsole/vconsole.conf among the tracked /etc targets. Neither exists. There is no config/vconsole/ directory and no links.conf line; /etc/vconsole.conf is now the untouched systemd fallback (every line commented out), systemd-vconsole-setup.service is masked in services.txt, and the keymap/consolefont hooks are gone from HOOKS in mkinitcpio.conf. An agent reading CLAUDE.md will look for a file that is not there.

Recommendation: drop both mentions from CLAUDE.md. If the Colemak-DH console keymap is still wanted on the TTY (see INFO-1), re-add config/vconsole/vconsole.conf with a links.conf line instead and unmask the unit.
CQ-3 low setup/install-programs:20-33

The install loop reads programs.txt on stdin (done < "$PROGRAMS_TXT") and calls yay -S --noconfirm "$package" inside it. yay/pacman inherit that stdin and can consume the remaining lines, so an arbitrary tail of the package list is silently skipped with no error - a classic read-loop stdin leak. On top of that, set -e means a single package that no longer exists in the repos or the AUR aborts the entire run, leaving the machine half-provisioned.

Recommendation: redirect the package manager's stdin (yay -S --noconfirm "$package" </dev/null) or read the file on a dedicated FD (while … done 3< "$file" with read -u 3), and let a failed package warn and continue instead of killing the run - the way install's own step functions already do.
CQ-4 low install:150-152,20-45

Adding an optional step is not a single-point change, which undercuts the point of the register_step registry. The registry supplies name, function and menu description, but the --flag must additionally be hand-added to the case list in the argument parser and, a third time, to the usage comment block at the top of the file. Forgetting the case entry produces unknown argument: --<step> for a step that is registered and does show up in the menu - encountered while adding --legion-conservation during this cycle.

Recommendation: drop the hardcoded flag list and match generically in the fallback branch - if ${STEP_FN[${1#--}]:-} is set, select that step; otherwise error. The help's step section can be generated from STEP_ORDER/STEP_DESC the same way, leaving register_step as the single source of truth.
UX-3 low install:316

The setup menu discards unrecognised input without a word: for n in $line; do [ -n "${idx[$n]:-}" ] && SELECTED[${idx[$n]}]=1; done. Typing 99, a step name instead of its number, or a comma-separated list (1,2 - the prompt asks for spaces) selects nothing, and setup then links the configs and reports success while having silently run zero of the steps the user asked for.

Recommendation: warn "unknown selection: $n" in the else branch, and re-prompt when the parsed selection ends up empty although the input was not.
CQ-5 low config/usrbin/vol_ctl:15,38

Two portability slips in a #!/bin/sh script, against the repo's own convention that only install and update_programs_list may use bashisms/GNU-isms. Line 38 uses grep -oP (PCRE - GNU grep only, and only when compiled with PCRE support). Line 15 parses pactl info by its English label (grep "Default Sink"), which breaks under a translated locale; it is safe here only because locale.gen generates en_US.UTF-8 and nothing else.

Recommendation: replace grep -oP '\d+%' with grep -o '[0-9]\+%' or an awk field pick, and pin the label lookup with LC_ALL=C pactl info.
CQ-6 low .gitignore · .claudeignore

Both files are commented in German, although CLAUDE.md fixes the comment/doc language at English for this repo - and the German is ASCII-mangled at that (ausschliessen, gehoeren), which the language rules in use here explicitly disallow. Both also point at Health-Report TRACK-2, a finding ID that no longer exists in this report, so the justification for the entries is now a dead reference.

Recommendation: translate both comment blocks to English and replace the TRACK-2 pointer with the reason itself ("never track secrets/runtime state that may end up under config/").
UX-4 low config/dwl/config.h:149

The screenshot binding Print → grim -g "$(slurp)" - | wl-copy is completely silent. There is no notification daemon in this setup (wob is a bar OSD, not a notifier), so nothing confirms that a screenshot was taken - the only way to find out is to paste. Cancelling slurp with Escape yields an empty geometry, whereupon grim fails equally silently. There is also no variant that writes the image to a file, so screenshots cannot be kept.

Recommendation: chain a short wob pulse on success (… && ~/.local/bin/osd 100) and add a second binding (Shift+Print) writing to ~/Pictures/screenshot-$(date +%F-%H%M%S).png.
INFO-1 info /etc/vconsole.conf · setup/services.txt:25

Consequence of the vconsole removal, recorded so it is a conscious state rather than a surprise: the TTY now runs the built-in us QWERTY keymap. On a Colemak-DH keyboard, the tty1 fallback shell you land in when dwl fails to start is therefore typed blind. The README argues correctly that this does not matter for login (autologin, no password on the VT) - but it does matter for repair. Not a bug, and cheap to accept; the point is that CLAUDE.md still claims the opposite (see DOC-3).

2 · What's already strong

strong install · setup/links.conf

The symlink manager is lean and genuinely safe. One mapping file, two columns, plain Bash with no dependencies (no Python, no dotbot, no Stow). The pipeline is explicitly phased - parse → validate → build → execute - and every command validates first, so a malformed links.conf aborts the run before anything is touched rather than silently skipping a line. Validation is fatal and precise (links.conf:<line>: <msg>) for missing targets, absolute or non-existent sources, duplicate expanded targets, targets outside the ~//etc//usr/local allowlist and globs that match nothing - with an optional keyword as the escape hatch for a legitimately empty glob. Real files are never overwritten unasked (--force backs up to .bak, with a timestamp suffix if a .bak already exists), unlink removes only symlinks that actually point at our own sources, and glob entries link per file so foreign files in ~/.local/bin survive. Verified live: validate green, 28/28 entries ok.

strong setup/services.txt · groups.txt · fonts.txt

Data lives in data files, not in the script. Units, groups and fonts are read into USER_UNITS/SYSTEM_UNITS/DISABLE_UNITS/MASK_UNITS, GROUP_LIST and FONT_PACKAGES by two small readers, with a read_list that uses a Bash nameref so the same six lines serve every list. services.txt even models intent rather than just names: the four scopes distinguish "enable", "we want this preset-on unit off" and "this one is static with no [Install], so it has to be masked" - and the comments explain why each distinction exists.

strong install - comment quality

The script comments the reasoning, not the syntax, and repeatedly pre-empts exactly the change a future maintainer would otherwise make and regret: why enable and not reenable (reenable's internal disable would delete our linked unit symlink), why a dead symlink in the parent path is removed before mkdir -p (it would otherwise fail with "File exists" and abort the whole run under set -e), why the getty drop-in is a real copy and not a symlink (systemd reads drop-ins before /home is mounted). That last one is the kind of thing you only learn by being bitten, and it is written down.

strong config/usrbin/update_programs_list · config/pacman/dotfiles-programs-list.hook

The package manifest maintains itself: a pacman PostTransaction hook regenerates setup/programs.txt after every pacman/yay transaction, so it cannot drift. The script writes atomically (mktemp in the target directory, trap cleanup, mv -f), so an interrupted transaction cannot leave a truncated manifest, and it chowns back to the owner when invoked as root by the hook. It also subtracts pacman -Qq base from pacman -Qe, keeping the list to what was actually chosen. Confirmed working during this review: installing waylock and removing hyprlock updated the file with no manual step.

strong config/nvim/

The Neovim config is the tidiest corner of the repo: one file per plugin under lua/plugins/ collected automatically by lazy.nvim, versions pinned in lazy-lock.json, tools provisioned declaratively (mason-lspconfig for servers, mason-tool-installer for formatters/linters), its own CLAUDE.md with its own verification commands, and formatting fixed by a checked-in stylua.toml. All modules parse, nvim --headless starts clean and stylua --check is green. The CLAUDE.md there also records the non-obvious traps (treesitter on main, the nvim-web-devicons preload mock, blink.cmp instead of nvim-cmp).

strong repo hygiene

No secrets in any tracked file (swept for password/token/key patterns and private-key headers). Every script carries the two-line SPDX/copyright header so the license travels with a single copied file, and third-party files (mpv scripts) keep their own. Shell style is consistent and defensive - set -eu throughout, dependency checks before use (command -v … || exit), battery detection that iterates /sys/class/power_supply/* instead of hardcoding BAT0, and numeric input validated before arithmetic in bat_check. No TODO/FIXME markers anywhere in our own code.

3 · Machine-specific - won't work as-is for another user

Personal dotfiles: the values below are hardcoded to this exact machine and user (a Lenovo Legion laptop, user leo, a specific LAN). They are not bugs - they are listed so a foreign user knows what to change before running ./install. Note that the automatic scrub described in earlier snapshots (--scrub / owner prompt) no longer exists, so all of this now has to be adjusted by hand.

machine-specific config/git/config

Hardcoded identity and signing key: name = Leonhard Weiler, email = leonhard-weiler@hotmail.com, signingkey = /home/leo/.ssh/id_ed25519.pub with commit.gpgsign = true. A foreign user commits under someone else's name and, worse, every commit fails outright because that absolute key path does not exist for them - with an error that does not obviously point at this file.

Adjust: set your own name/email and point signingkey at your own key - as ~/.ssh/… rather than /home/leo/…, which would at least make the path user-agnostic.
machine-specific config/usrbin/restic-backup

The backup target defaults to a private NAS (sftp:Leo@192.168.178.23:D:\ResticBackup) and the backed-up paths are fixed to ~/files/projects and ~/files/school. The repository is at least overridable via RESTIC_REPOSITORY; the paths are not. The restic forget --prune branch behind the y/n prompt is destructive, and under set -eu a non-interactive run simply exits at the read instead of pruning - safe, but only by accident.

Adjust: export RESTIC_REPOSITORY and lift the two backup paths into overridable variables.
machine-specific config/dwl/config.h · install (legion step)

monrules[] pins the internal panel by name (eDP-1) at scale 1.6 - on other hardware the output is named differently and falls through to the NULL catch-all at scale 1.0. The --legion-conservation step writes to the ideapad_acpi sysfs path (VPC2004:00), which only exists on Lenovo Legion/IdeaPad hardware - it detects this and skips itself elsewhere, so it is harmless, just inert.

Adjust: set your own output name and scale in monrules[] and rebuild with ./install --dwl.
machine-specific config/systemd-system/getty@tty1.service.d/autologin.conf · config/keepassxc/keepassxc.ini

The autologin drop-in hardcodes the username leo in its agetty --autologin line - on a foreign machine tty1 then tries to log in a user that does not exist. KeePassXC's BackupFilePathPattern likewise points at /home/leo/.config/keepassxc/….

Adjust: replace the username in the drop-in before running ./install --getty-autologin, and repoint the KeePassXC backup pattern at your own home.

4 · Deliberate decisions

deliberate config/dwl/

dwl is configured at compile time: config.h is the source of truth and is deliberately not symlinked - it is compiled into the binary, so ./install --dwl is the apply step. The suckless trade-off is accepted knowingly: no runtime reload, but no config parser and no runtime IPC either. dwl itself is pinned to v0.8 and cloned at build time rather than vendored; only the built binary is installed.

deliberate config/bash/.bash_profile · --getty-autologin

No display manager. getty@tty1 autologins via an agetty --autologin drop-in and ~/.bash_profile execs dwl on tty1 - guarded so it only fires on tty1 and only when no Wayland session is already running. The drop-in is deployed as a real copy, not a symlink, because systemd reads unit drop-ins before /home is mounted, when a symlink into the repo would still be dangling. That constraint is the reason the step exists at all.

deliberate config/dwl/config.h (lockcmd)

The screen locker is waylock, replacing hyprlock. waylock has no config file - it is configured purely through CLI flags, so the whole configuration is the lockcmd[] array. The cost is accepted and written down in the README: waylock only ever paints one solid colour, so the screenshot background, blur, the input field (size, outline, rounding, placeholder, font), hide_cursor and fail_timeout are gone. In exchange the dependency drops from the Hyprland stack (hyprlang/hyprutils/hyprgraphics/sdbus-cpp/pugixml) to a single ISC binary.

deliberate config/nvim/lua/plugins/conform.lua

prettier, not prettierd. The daemon variant deliberately outlives Neovim and stayed resident at ~60 MB after the last buffer was closed. css moved to biome (which already formats the JS/TS/JSON side); html and markdown stay on plain prettier because biome's html formatter is experimental and off by default and biome has no markdown formatter at all - both verified against biome 2.5.2 during this review. The trade is a few hundred ms per format against no permanently resident process.

deliberate install (legion step)

The battery conservation mode is a one-shot setup step, not a boot-time unit. The ideapad_acpi driver writes the flag through to the embedded controller, which keeps it across reboots and power-offs - verified by disabling and removing the former legion-conservation.service and re-reading the sysfs value. A service firing at every multi-user.target to write a value that was already there was pure boot-time cost.

deliberate config/wbg/ · config/wallpaper/

The wallpaper feature is stood down, not deleted: nothing is installed, linked or autostarted, but the build-wbg script and the picture set stay in the repo, with the disabled links.conf lines and the disabled autostart[] line commented in place and the revival path spelled out in CLAUDE.md. Reversible by uncommenting three lines plus ./install --wbg --dwl.

deliberate no test framework

There is no test suite and no CI. For a config repo whose "logic" is one Bash script, the gates are the cheap ones: bash -n/sh -n for scripts, luajit -bl and a headless Neovim start for the Lua, stylua --check for formatting, and above all ./install validate plus ./install -n as a true dry run for the link map. Deliberate, and adequate at this size - the one thing it does not cover is the /etc side, which only shows up on a real machine.

deliberate .gitignore · .claudeignore

The KeePassXC database (*.kdbx) is git-ignored and the whole config/keepassxc/ directory is additionally excluded from agent access via .claudeignore, while the (secret-free) keepassxc.ini stays tracked. The Claude config is tracked selectively for the same reason: settings.json and skills/ only, never .claude.json, sessions, history or cache.
